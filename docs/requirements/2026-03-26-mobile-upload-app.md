# mo-gallery 移动上传 App 需求冻结文档

## 背景

当前仓库已经具备完整的 Web 侧内容管理能力，包括管理员登录、照片上传、相册管理、故事管理、去重校验、草稿缓存和公开展示。现有实现更偏向桌面 Web 后台与 Web 创作场景，在手机端完成大量相册选图、批量上传和弱网恢复时，交互成本和稳定性都不理想。

本次需求聚焦于为项目补一个仅供站点所有者本人使用的移动端 App，目标不是替代现有 Web 站，而是补齐高频移动创作入口，优先解决“从手机相册批量选图并可靠上传到现有相册 / 故事体系”。

## 目标

构建一个 `Flutter` 移动 App 第一阶段版本，用于：

- 仅本人登录使用
- 从手机相册多选图片
- 批量上传图片到现有后端
- 默认投递到最近使用的目标
- 允许在上传前临时修改目标
- 一个批次可同时关联多个目标
  - 可关联 0 个或多个相册
  - 可关联 0 个或多个故事
- 在失败、弱网、应用退出后尽量自动恢复未完成上传

## 交付物

第一阶段交付物包括：

- 一份面向当前仓库的移动 App 方案与实施计划
- 基于现有 API 的移动端接入设计
- 必要的后端补强方案定义
- 明确的第一阶段范围、验收标准与非目标

本次冻结文档不要求立即完成代码实现，但后续实现必须以本文档为准。

## 用户与使用模式

### 目标用户

- 唯一用户：站点所有者 / 管理员本人

### 使用场景

- 在手机中从系统相册多选一批照片
- 快速投递到最近常用相册或故事
- 在上传前临时更改目标集合
- 上传过程里切到后台、网络抖动、返回 App 后仍能继续推进
- 上传后能在现有 Web 后台和前台体系中直接看到结果

## 约束条件

### 产品约束

- 第一阶段只聚焦“快速传图”
- 不要求在 App 内完成复杂写作或完整后台管理
- 不考虑多人账户、多角色权限和开放注册

### 技术约束

- 现有主仓库技术栈为 `Next.js + Hono + Prisma`
- 现有后端已提供：
  - 登录与 JWT 鉴权
  - 管理员相册列表
  - 管理员故事列表
  - 单图上传
  - 文件哈希重复检测
  - 照片关联相册
  - 照片关联故事
- 当前上传接口为单文件表单上传，尚未具备真正的分片断点续传能力

### 范围约束

- 第一阶段“退出后继续”定义为：
  - 上传队列状态持久化
  - 应用重启后自动恢复待上传任务
  - 对未完成文件自动重试
  - 借助文件哈希与服务端重复检测避免重复入库
- 第一阶段不强制要求字节级分片续传

## 验收标准

以下能力全部满足，视为第一阶段需求达标：

1. 管理员能够在移动 App 中完成登录，并保持可用会话
2. 能从系统相册一次选择多张图片并形成上传批次
3. 进入上传页面时自动带出最近使用的目标集合
4. 用户可在上传前临时调整关联的相册与故事集合
5. 上传成功后的照片能够自动挂接到所选相册与故事
6. 同一批次允许同时关联多个相册、多个故事，也允许为空集合
7. 网络抖动或单张失败时，任务会自动重试
8. 关闭 App 后重新进入，未完成任务可继续推进，不要求重新手动选完整批次
9. 已上传完成的文件不会因恢复流程造成重复照片记录
10. Web 端现有相册、故事、照片能力不发生回归

## 非目标

第一阶段明确不做以下事项：

- 完整复刻 Web 后台管理功能
- 在 App 中进行复杂故事写作和排版
- 多人登录与协作上传
- 推送通知体系
- 原生相机深度控制
- 云端分片断点续传
- 公共用户上传入口

## 现有能力盘点

### 已存在且可复用的能力

- `src/lib/api/auth.ts`
  - 用户名密码登录，返回 JWT
- `src/lib/api/albums.ts`
  - 获取管理员相册
  - 向相册添加照片
- `src/lib/api/stories.ts`
  - 获取管理员故事
  - 向故事添加照片
- `src/lib/api/photos.ts`
  - 单图上传
  - 文件哈希重复检测
  - 批量重复检测
- `hono/photos.ts`
  - 服务端单图上传与哈希去重
- `src/lib/client-db.ts`
  - Web 侧 IndexedDB 草稿持久化，可作为移动端持久队列思路参考

### 现有不足

- 没有移动端专用的“最近目标”配置存储
- 上传后关联相册 / 故事需要 App 侧多次串行调用
- 没有上传队列的服务端可观测性记录
- 没有真正的断点续传协议

## 冻结的产品决策

### 平台决策

- 移动端技术选型冻结为 `Flutter`
- Web 站继续作为公开浏览与后台管理主入口
- App 只承担移动高频上传工作流

### 上传目标决策

- 默认使用最近一次成功上传时的目标集合
- 用户可以在本次上传前修改目标集合
- 目标集合支持：
  - 0 个或多个相册
  - 0 个或多个故事
- 一批照片的目标集合对整批生效

### 恢复语义决策

- 第一阶段实现“任务级恢复”，不承诺“字节级恢复”
- App 必须持久化本地上传队列
- 对处于未完成状态的任务，应用恢复后自动重试
- 服务端利用 `file_hash` 与重复检测保证幂等

## 推荐方案

推荐采用“`Web/后端 保持不变为主，Flutter 新建独立移动客户端，后端做少量移动补强`”方案。

### 原因

- 当前仓库已经具备上传与挂接主链路，适合复用现有领域模型与接口
- 该需求对移动端相册访问、后台恢复、网络重试、状态持久化依赖较高，`Flutter` 在移动端工具型 App 上的确定性更强
- 需求重心是“上传可靠性”，不是“复用现有 Web UI”

## 需求边界中的关键假设

- 本人可以接受第一阶段使用管理员账户直接登录
- 本人接受第一阶段的“继续上传”语义为“自动恢复任务并重新发起未完成文件”，而不是底层字节流续传
- 本人接受 App 与 Web 分离，前者专注上传效率，后者承担整理和深度管理

## 风险

### 高风险

- 如果移动端在系统回收后无法重新拿到本地资源句柄，恢复体验会受影响
- 现有后端接口偏“单图上传 + 后续挂接”，批量任务量大时移动端编排复杂
- 若目标集合变更多、附件更多，失败回滚和局部成功提示需要设计清楚

### 中风险

- 移动端图片压缩、原图保留、EXIF 保留策略需要和现有服务端处理保持一致
- JWT 长期会话策略如果太弱，恢复队列时容易中断

### 低风险

- 现有相册 / 故事查询与挂接能力已经存在，复用成本低

## 需求批准条件

若进入实现阶段，以下内容视为已获批准：

- 使用 `Flutter` 构建独立移动上传 App
- 第一阶段主目标为自用批量传图
- 允许多相册、多故事同时关联
- 实现任务级恢复与自动重试
- 后端可进行必要但尽量克制的补强
