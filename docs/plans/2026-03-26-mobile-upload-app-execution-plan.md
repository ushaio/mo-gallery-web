# mo-gallery 移动上传 App 执行计划

## 计划摘要

基于已冻结需求，采用 `L` 级治理执行策略：先完成设计与接口补强，再进行单客户端实现与集成验证。本任务暂不进入代码实现，当前交付执行计划与阶段边界，后续需在用户确认后进入开发。

## 内部执行等级

- 等级：`L`
- 原因：
  - 需求复杂度中等偏高，但写范围清晰
  - 工作以设计、接口梳理、移动端架构与可靠性策略为主
  - 当前不需要并行子代理实施多个独立代码改动面

## 当前仓库能力结论

现有仓库已经具备以下基础链路：

- 管理员登录与 JWT 鉴权
- 获取管理员相册和故事
- 单图上传
- 文件哈希去重
- 上传后向相册 / 故事挂接照片

因此第一阶段应优先复用已有服务接口，而不是先改造为全新上传后端。

## 推荐总体架构

### 客户端

新增独立 `Flutter` App，职责包括：

- 登录与会话缓存
- 相册多选
- 批量上传队列管理
- 最近目标集合缓存
- 自动重试
- 应用退出后恢复队列

### 服务端

继续使用现有 `Next.js + Hono + Prisma` 仓库，职责包括：

- 账号鉴权
- 文件去重
- 图片入库
- 图片与相册 / 故事关系建立
- 可选的移动端偏好存储与批量挂接增强

### 数据流

1. App 登录并缓存 JWT
2. 拉取管理员相册 / 故事列表
3. 读取最近目标集合
4. 系统相册多选形成本地队列
5. 逐项计算文件哈希并做批量重复检测
6. 对非重复文件执行上传
7. 上传完成后将照片批量挂到所选相册与故事
8. 更新最近目标集合
9. 将失败任务留在本地队列并自动重试

## 方案对比

### 方案 A：Tauri 移动端直接复用 Web UI

优点：

- 复用 React 代码较多

缺点：

- 移动端相册访问、后台恢复、弱网可靠性控制力不足
- Web 编辑 UI 与移动上传工具的交互目标不一致
- 首期最关键能力不是页面复用，而是任务可靠性

结论：

- 不推荐作为主方案

### 方案 B：Flutter 独立 App，直接复用现有 API

优点：

- 开发路径最短
- 第一阶段即可上线
- 后端改动最少

缺点：

- 最近目标、批量挂接、任务可观测性需要客户端多做编排

结论：

- 推荐作为第一阶段起步方案

### 方案 C：Flutter 独立 App + 后端增加移动专用接口

优点：

- 交互和可靠性更整洁
- 有利于后续扩展写作与移动专属功能

缺点：

- 首期接口设计成本更高

结论：

- 推荐作为第一阶段落地时的“有限补强版”

## 最终推荐

采用“`Flutter App + 复用现有 API + 后端少量补强`”。

后端补强仅限于直接提高移动上传体验的部分，不引入重型分片上传体系。

## 阶段划分

### Phase 0：接口与领域梳理

目标：

- 明确移动端复用哪些现有接口
- 明确哪些能力需要补强

输出：

- 移动端 API 对照表
- 上传任务状态机
- 最近目标集合的数据定义

### Phase 1：服务端补强

目标：

- 降低移动端编排复杂度
- 提供最少但关键的移动能力

建议新增或调整内容：

- 新增移动偏好接口
  - `GET /api/admin/mobile/preferences`
  - `PUT /api/admin/mobile/preferences`
  - 存储最近使用的相册 ID 集合和故事 ID 集合
- 可选新增批量挂接接口
  - 方案 1：`POST /api/admin/mobile/upload-links`
  - 方案 2：保持现有接口不变，由 App 分别调用 `addPhotosToAlbum` / `addPhotosToStory`
- 明确重复文件响应规范，保证客户端可稳定识别“已存在”

### Phase 2：Flutter App MVP

目标：

- 跑通核心闭环

必须包含：

- 登录页
- 目标选择页 / 上传页
- 相册多选接入
- 本地上传队列
- 自动重试
- 恢复未完成任务
- 上传结果页

### Phase 3：稳定性增强

目标：

- 提升弱网和长任务可用性

建议包含：

- 并发数控制
- 指数退避重试
- 局部成功提示
- token 失效重登恢复
- 上传结果回放与错误分组

## 客户端状态机设计

每个文件任务建议使用以下状态：

- `queued`
- `hashing`
- `duplicate_checking`
- `uploading`
- `uploaded`
- `linking_targets`
- `completed`
- `failed_retryable`
- `failed_terminal`

恢复逻辑：

- `queued` / `hashing` / `duplicate_checking` / `uploading` / `linking_targets`
  - App 重启后统一视为待恢复任务
- `uploaded`
  - 若已拿到 `photoId` 但未挂目标，只补执行挂接
- `completed`
  - 不再重复执行

## 可靠性策略

### 第一阶段可靠性定义

- 自动重试
- 队列持久化
- 应用重启后恢复未完成任务
- 服务端去重避免重复照片

### 不承诺的能力

- 真正的字节级断点续传
- 无网络条件下跨设备同步队列

### 推荐实现

- 本地持久化上传队列与任务快照
- 每个任务保存：
  - 本地资源标识
  - 文件名
  - 哈希值
  - 上传状态
  - 服务端 `photoId`
  - 重试次数
  - 错误摘要
  - 目标相册 ID 列表
  - 目标故事 ID 列表
- 若资源重新解析失败，则标记为需要人工重选

## 后端补强建议

### 必做

- 统一重复上传错误码与返回体，保证客户端可将重复视为“可恢复已存在”
- 评估是否需要单独的移动偏好存储

### 建议做

- 增加一个“批量挂接多个目标”的服务接口，降低移动端重复请求量
- 在服务端记录上传来源 `web | mobile`，便于后续排查

### 暂不做

- 分片断点续传
- 独立上传任务表
- 推送通知

## 验证策略

### 服务端验证

- `pnpm run lint`
- `pnpm run build`
- 若涉及 Prisma 变更：
  - `pnpm run prisma:generate`
  - 必要时迁移检查

### 移动端验证

- Android 真机或模拟器：
  - 选取 20 张图片批量上传
  - 中途断网后自动重试
  - 上传中杀进程后恢复
  - 重复上传同一批次不产生重复照片
- 目标挂接验证：
  - 0 目标
  - 仅相册
  - 仅故事
  - 多相册 + 多故事

### 回归验证

- Web 端照片上传流程不受影响
- Web 端相册与故事展示正常

## 回滚规则

- 若后端补强导致现有 Web 上传回归，优先回滚新增移动接口改动
- 数据库若新增偏好表，迁移需可逆或可安全废弃
- 移动端独立仓库发布失败不影响现有 Web 可用性

## 风险与缓解

### 风险 1：资源恢复失败

缓解：

- 优先保存系统资源 ID 与必要元数据
- 对不可恢复资源给出清晰提示并允许用户手动重选

### 风险 2：上传后挂接失败导致半成功

缓解：

- 任务状态拆分为“上传成功”和“挂接成功”
- 恢复流程只重做未完成挂接部分

### 风险 3：重复检测与真实文件不一致

缓解：

- 客户端统一计算哈希
- 服务端将 `409 DUPLICATE_PHOTO` 视为已存在资源返回可解析信息

## 执行顺序

1. 冻结需求与计划
2. 设计移动端状态机与 API 合约
3. 确认是否需要后端移动偏好接口
4. 实现后端补强
5. 搭建 Flutter MVP
6. 完成上传恢复链路
7. 做端到端验证

## 进入实现前的暂停点

当前计划完成后，需等待用户确认是否进入代码实现。未经确认，不直接开始创建 Flutter 工程或修改后端逻辑。
